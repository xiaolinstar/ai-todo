# ai-todo v0.1.3 版本规划

日期：2026-06-03
发布目标：微信小程序体验版 / 内测体验修订版

## 版本定位

`0.1.3` 是 `0.1.2` 发布后的体验修订版本。目标是继续围绕真实内测反馈做小步优化，不引入高风险隐私能力，不扩大数据模型，优先让提醒、日历和应用内联系人创建更顺手。

本版本继续保持微信小程序客户端、FastAPI 服务端和 CLI 一致发布。即使只有小程序发生改动，也使用同一个 `v0.1.3` tag 作为发布锚点。

## 前置条件

开始 `0.1.3` 开发前，应先完成：

- 微信小程序 `0.1.2` 已提交审核或发布体验版。
- 服务端已可使用 `release_tag=v0.1.2` 完成手动 CD。
- 生产 API 能通过：
  - `/v1/health`
  - `/v1/health/db`
  - `releaseTag=v0.1.2`
- 收集至少一轮真实使用反馈：
  - 创建提醒是否足够快；
  - 创建日历是否清晰；
  - 备注 / 描述折叠是否自然；
  - 隐私授权登录是否还有异常。

## 版本范围

### P0：0.1.2 发布后回归

- 跑一遍发布后 smoke test：
  - 微信登录；
  - 隐私保护提示；
  - 个人资料修改；
  - 创建提醒；
  - 完成提醒；
  - 左滑删除提醒；
  - 创建日历；
  - CLI `whoami` / `today` / `reminder list` / `calendar list`。
- 修复 `0.1.2` 审核或体验版中发现的阻塞问题。
- 确认 `.env`、CD、health 元信息仍与发布 tag 对齐。

验收标准：

- `0.1.2` 线上版本可以稳定使用。
- 没有影响登录、创建、删除、同步的阻塞问题。

### P1：提醒体验小修

- 评估是否新增提醒详情页：
  - 当前提醒列表已经可以完成和删除；
  - 详情页可用于展示完整标题、备注、截止时间、联系人；
  - 暂不做复杂编辑，避免范围膨胀。
- 评估提醒备注展示方式：
  - 列表中不展示大段备注；
  - 详情页或轻量弹层展示完整备注。
- 继续观察左滑删除：
  - 是否需要撤销删除；
  - 是否需要删除后 toast 文案优化；
  - 暂不恢复“已删除列表”。

验收标准：

- 高频提醒操作不增加额外步骤。
- 用户能查看完整提醒信息。
- 删除仍保持二次确认，避免误删。

### P1：创建表单体验继续打磨

- 复查备注 / 描述折叠箭头视觉：
  - 是否足够像可展开控件；
  - 是否需要换成右箭头 / 下箭头系统样式；
  - 是否需要点击整行有轻微反馈。
- 复查日期时间合并行：
  - 小屏设备是否拥挤；
  - 日期、时间是否容易点击；
  - 是否需要把显示格式从 `YYYY-MM-DD` 缩短为 `6月3日`。
- 评估是否增加默认快捷时间：
  - 今天；
  - 明天；
  - 18:00；
  - 20:00。

验收标准：

- iPhone 常见宽度下不拥挤、不换行异常。
- 用户能在 10 秒内创建一个简单提醒。

### P1：联系人创建体验

- 将通讯录页内嵌创建表单改为独立“新建联系人”页面：
  - 从底部 `+` 的“添加联系人”进入；
  - 页面风格与新建提醒、新建日历保持一致。
- 解决当前底部弹层中按钮显示不全的问题。
- 字段保持应用内联系人边界：
  - 姓名；
  - 邮箱；
  - 电话；
  - 公司；
  - 职位；
  - 备注。
- 表单按信息优先级拆分：
  - 基础联系方式：姓名、邮箱、电话；
  - 补充信息：公司、职位、备注。
- CLI 同步支持联系人创建 / 更新中的公司、职位和备注字段。

验收标准：

- 新建联系人页面不遮挡、不露边，按钮完整显示。
- 添加联系人成功后返回通讯录页并刷新列表。
- 不读取或写入手机系统通讯录。

### P2：日历视图可读性

- 优化日历列表展示：
  - 时间范围更清晰；
  - 地点、联系人展示更克制；
  - 空状态文案保持简洁。
- 评估是否增加日历详情页：
  - 展示标题、时间、地点、描述、联系人；
  - 暂不做重复日历、冲突检测或系统日历写入。

验收标准：

- 今天日历视图可快速扫读。
- 不引入系统日历权限。

### P2：CLI / 发布验收

- 将常用生产 smoke 命令整理成固定脚本或文档段落。
- 评估 CLI 是否需要 `smoke` 命令：
  - `ai-todo smoke --api <url>`；
  - 或继续保持文档化手工命令。
- 保持 PAT 明文只在创建时返回。

验收标准：

- 发布后 5 分钟内能完成服务端和 CLI 验收。
- 测试数据可清理。

## 隐私与合规门禁

`0.1.3` 默认不新增隐私类目。继续允许：

- 微信登录标识；
- 微信头像、昵称；
- 用户主动输入的提醒、日历、应用内联系人；
- 用户在输入框中自行粘贴的文本；
- CLI 访问令牌；
- 订阅消息授权。

如果出现以下能力，必须先暂停实现，更新 [privacy-compliance.md](../privacy-compliance.md)，并完成微信公众平台报备：

- `getPhoneNumber`
- `chooseImage` / `chooseMedia`
- `getLocation` / `chooseLocation`
- `getClipboardData`
- 读取或写入手机系统通讯录
- 读取或写入微信/手机系统日历
- 上传头像、图片、附件、语音或视频到服务端
- 接入广告、统计或第三方 SDK

发布前固定检查：

```bash
rg "getPhoneNumber|chooseImage|chooseMedia|getLocation|chooseLocation|getClipboardData|getUserProfile|requestSubscribeMessage|chooseAvatar|type=\"nickname\"|onNeedPrivacyAuthorization|requirePrivacyAuthorize|agreePrivacyAuthorization" apps/miniapp/miniprogram apps/miniapp/types
```

检查结果必须与隐私报备清单一致。

## 明确不做

`0.1.3` 暂不做：

- 手机号登录或绑定。
- 定位、地图选点或地址能力。
- 相册、摄像头、文件上传。
- 手机系统通讯录读取或写入。
- 微信/手机系统日历写入。
- 复杂重复提醒。
- 复杂重复日历。
- 蓝绿 / 灰度发布完整实现。
- MCP Server。

这些能力进入 `0.2.0` 或后续版本重新评估。

## 建议实施顺序

1. 完成 `0.1.2` 审核 / 体验版 / 服务端 CD 验收。
2. 收集真实使用反馈，优先记录阻塞问题。
3. 修复 `0.1.2` 回归问题。
4. 做提醒详情或创建表单小修，二选一优先。
5. 跑隐私接口检查、`pnpm check:wechat`、`pnpm test:api`。
6. 生成 `v0.1.3` 发布说明。
7. 打 `v0.1.3` tag，小程序提交版本，服务端手动 CD。

## 发布备注草案

`0.1.3` 是 ai-todo 内测体验修订版，重点修复 `0.1.2` 体验版反馈，并继续优化提醒、日历和应用内联系人创建体验。本版本保持微信小程序客户端、FastAPI 服务端和 CLI 一致发布，不新增手机号、定位、相册、摄像头、麦克风、系统通讯录等隐私能力。
