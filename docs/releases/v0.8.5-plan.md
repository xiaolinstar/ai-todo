日期：2026-06-25
状态：**已验收 / 待发布**

### 产品判断

最近使用频率不高，下一版不适合做重功能。优先做「再次打开时不用重新理解系统」的修补：

- 表单详情页要能读完整信息，尤其是提醒标题。
- CLI / Agent 可能写入多个联系人，小程序不能在查看或保存时悄悄丢掉后续联系人。
- 联系人能力先做一致与保真，不急着做复杂的多人选择器。

## 问题陈述

| # | 用户现象 | 当前实现 | 风险 |
|---|----------|----------|------|
| 1 | 小程序创建/编辑提醒和日程时只能选 1 个联系人，但 CLI 可以传多个 `--contact` | API 数据模型支持 `contactIds[]`；CLI `readRepeatedFlag(argv, "--contact")` 支持多个；小程序编辑页只读取 `contacts[0]` 并保存为单元素数组 | CLI 创建的多联系人事项，一旦在小程序打开并保存，会丢失第 2 个及之后联系人 |
| 2 | 提醒事项标题在列表里显示不全，点进去后仍看不到完整标题 | 详情页标题使用右对齐单行 `<input class="form-input">`；全局表单输入是单行展示 | 低频使用时需要靠标题恢复上下文，详情页不能承担「读完整」职责 |
| 3 | 日程详情页也有同类标题输入模式 | 日程编辑页同样使用单行标题 input | 可顺手修复，避免提醒修了、日程仍不一致 |

## 产品决策

| # | 问题 | 默认决策 |
|---|------|----------|
| 1 | 小程序是否立即支持多人选择？ | **本版先做保真 + 展示多联系人**；新建仍可保持单选，编辑已存在多联系人时必须展示全部并保存全部 |
| 2 | 小程序保存时如何处理 CLI 多联系人？ | 默认不丢失。若用户没有显式移除联系人，保存时带回原完整 `contactIds` |
| 3 | 是否做多选联系人 picker？ | P1 可选；若实现成本低，再把 picker 扩成可勾选多联系人。否则留到 v0.9 前 |
| 4 | 标题如何完整显示？ | 详情页标题改为多行 `textarea` 或「完整标题阅读区 + 编辑输入」。默认采用多行 `textarea`，最大 80 字不变 |
| 5 | 列表标题是否也展开？ | 不展开。列表继续单行省略，保证扫描效率；详情页负责完整阅读 |

## P0 — 联系人保真

### REQ-0.1 小程序编辑页保留多联系人

**目标**：任何由 CLI / Agent / API 写入的多联系人提醒或日程，在小程序查看和保存后联系人不减少。

**范围**：

- 提醒编辑页：`apps/miniapp/miniprogram/pages/reminder-edit/*`
- 日程编辑页：`apps/miniapp/miniprogram/pages/event-edit/*`

**期望行为**：

1. 加载详情时保存完整 `contacts[]`，而不是只取 `contacts[0]`。
2. 页面上展示联系人汇总：
   - 0 个：`选择`
   - 1 个：联系人姓名
   - 多个：`张三、李四等 3 人` 或换行 chip 列表
3. 未修改联系人时，提交 payload 使用原完整联系人 id 列表。
4. 用户点「移除联系人」时清空全部联系人；若保留单选 picker，选择新联系人表示用该联系人替换全部联系人。
5. 保存后再次打开，联系人数量与保存前一致。

**验收**：

- [ ] CLI 创建提醒：`--contact a --contact b`，小程序打开后显示 2 人。
- [ ] 不改联系人直接保存，API 返回仍包含 2 人。
- [ ] 小程序移除联系人后保存，API 返回 0 人。
- [ ] 日程编辑页同样通过上述 3 条。

## P0 — 详情页标题完整可读

### REQ-1.1 提醒详情标题多行显示

**目标**：列表可以省略，详情页必须能读完整标题。

**范围**：

- `apps/miniapp/miniprogram/pages/reminder-edit/reminder-edit.wxml`
- `apps/miniapp/miniprogram/pages/reminder-edit/reminder-edit.scss`
- 可选抽取全局样式：`apps/miniapp/miniprogram/app.scss`

**期望行为**：

1. 标题控件从单行右对齐 input 改为多行标题输入：
   - 保持 `maxlength="80"`
   - 支持自动增高或固定最小高度
   - 文本左对齐，长标题自然换行
2. 标题为空校验保持不变。
3. 小屏下标题不挤压「标题」标签，不与下一行重叠。
4. 备注仍在独立折叠区，不把标题长文本塞进备注。

**验收**：

- [ ] 80 字标题进入提醒详情页能完整读到。
- [ ] 修改标题后保存成功，列表仍按单行省略展示。
- [ ] iPhone 窄屏宽度下标题区域不溢出、不遮挡截止时间行。

### REQ-1.2 日程详情标题同步修复

日程详情页标题使用同一套多行样式，避免同一类信息在提醒/日程之间体验不一致。

**验收**：

- [ ] 长日程标题进入详情页能完整读到。
- [ ] 修改保存后日历列表/详情数据正确。

## P1 — 可选增强

### REQ-2.1 多选联系人 picker

若 P0 做完后仍有余量，将 `contact-picker` 扩展为可配置单选/多选：

- `mode=single|multiple`
- 多选模式显示勾选态与底部确认按钮
- 提醒/日程新建、编辑页都可维护多个联系人

**不做也可发布**：P0 的保真修复已经解决 CLI 与小程序之间的数据损失。

### REQ-2.2 Agent/Skill 约束

更新 `skills/ai-todo/SKILL.md` 联系人写入建议：

- 多联系人可以写入，但提醒/日程标题不要堆人名。
- 如果事项实际上只有一个主要对象，优先只关联主联系人，其余上下文放 notes。
- 当用户语义明确是多人会议/多人协作时，继续使用多个 `--contact`。

## 非目标

- 不改 API 数据模型或数据库迁移。
- 不限制 CLI 的多个 `--contact` 能力。
- 不做提醒标签、优先级、搜索筛选。
- 不做复杂的通讯录分组、最近联系人、智能推荐。

---

## 建议实施顺序

1. 提醒/日程编辑页数据结构从 `selectedContact` 改为 `selectedContacts`，先保证保存不丢多联系人。
2. 增加联系人展示文案或 chip 列表。
3. 标题控件改为多行，并在提醒/日程详情页复用样式。
4. 手工验收 CLI 多联系人写入 → 小程序编辑保存 → CLI/API 再读。
5. 跑门禁：`pnpm check:wechat`；若只改小程序，无需 API bump。
