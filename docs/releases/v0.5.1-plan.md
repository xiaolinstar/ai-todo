## 产品结论：共用模板 #15788

**不申请新模板**（本批 P0）。理由：

- 三个语义槽位足够：标题、时间点、补充说明
- 同一 `template_id` → 用户只需熟悉一种消息卡片样式
- 订阅配额按模板计，共用模板不增加用户授权复杂度

当前模板 keyword（已对接 worker）：

| 公众平台展示名 | keyword | 类型 |
|---------------|---------|------|
| 事项主题 | `thing23` | thing |
| 截止日期 | `time2` | time |
| 备注 | `thing13` | thing |

## 字段映射与「是否需要打磨」

### 推荐映射（v0.5.1 采用）

| keyword | 提醒（现状） | 日程（新增） |
|---------|-------------|-------------|
| `thing23` | `reminder.title` | `calendar_event.title` |
| `time2` | `remind_at` ?? `due_at` | **`start_at`**（账户时区格式化） |
| `thing13` | `notes` 或「点击查看提醒详情」 | 见下方 **备注拼装规则** |

**备注拼装规则（`thing13`，≤20 字）：**

1. 有 `location` → 优先 `location`（截断）
2. 否则有 `description` → 取 description 前 20 字
3. 否则固定文案：**「点击查看日程详情」**
4. **可选增强（P1）**：若有 `end_at`，且 location/description 为空，用 `14:00-15:00` 形式填入 `thing13`（开始时间仍在 `time2`）

### 展示名「截止日期」vs 日程「开始时间」

| 方案 | 说明 | 本批 |
|------|------|------|
| **A. 不改模板，语义上容忍** | 消息里第二行仍显示「截止日期: 2026年6月6日 14:00」，实际表示日程开始时刻 | **P0 采用** |
| **B. 公众平台改展示名** | 若模板允许仅改标签为「时间」而不变 keyword，可减轻歧义 | 可选，**非阻塞** |
| **C. 申请新模板** | 字段改为「事项主题 / 时间 / 备注」，keyword 会变，需改 worker + 重新配置 env | **不做**（除非 A 引发大量误解） |

**结论：** v0.5.1 用方案 A；内测观察用户反馈，再决定是否走 C。

### 触发时刻（scheduled_at）

| 方案 | 行为 | 本批 |
|------|------|------|
| **到点开始** | `scheduled_at = start_at` | **P0（唯一策略；用户改开始时间即改推送时刻）** |
| ~~**提前 N 分钟**~~ | ~~start_at - N~~ | **不做（已确认）** |
| **到结束提醒** | `scheduled_at = end_at` | 不做（与「开始日程」心智不符） |

提醒与日程共用 worker 扫描逻辑，仅 `scheduled_at` 来源不同。

## 与 v0.5.0 的差异（架构）

```text
创建/改期日程
  → requestSubscribeMessage（同一 template_id）
  → POST /v1/notifications/wechat/subscription-result
       target_type=calendar_event, template_key=calendar_event_start
  → notification_deliveries (scheduled_at = start_at)
  → worker 到期发送（thing23/time2/thing13 与提醒共用映射函数）
  → 用户点击 → calendar?eventId= → event-edit
```

订阅配额：微信按 **template_id** 计次，提醒与日程各授权一次各消耗 1 次，符合预期。
