# ai-todo v0.1.2 版本规划

日期：2026-06-02
发布目标：微信小程序体验版 / 内测稳定修订版

## 版本定位

`0.1.2` 是 `0.1.1` 发布后的内测打磨版本。目标不是新增大功能，而是让真实用户内测更顺、更可验证、更合规：

- 保持微信小程序客户端、FastAPI 服务端、CLI 的版本节奏一致。
- 补齐个人资料设置入口和账号体验小问题。
- 提升生产发布后的可观测性和验收效率。
- 收敛剪切板相关体验，保持用户自行粘贴的自然输入方式和明确隐私门禁。

## 前置条件

开始 `0.1.2` 开发前，应先完成：

- `v0.1.1` 服务端使用 `release_tag=v0.1.1` 完成手动 CD。
- 生产环境验证：
  - `https://wodi.games/v1/health`
  - `https://wodi.games/v1/health/db`
  - 微信登录
  - CLI PAT 登录
  - 提醒、日历、联系人基础 CRUD smoke test
- 微信小程序 `0.1.1` 已发布或至少完成体验版验收。
- 微信小程序隐私报备已按 [privacy-compliance.md](../privacy-compliance.md) 完成。

## 版本范围

### P0：发布与验收闭环

- 确认 `CD` workflow 使用 `release_tag` 作为日常发布入口。
- 在发布手册中明确：
  - `Use workflow from` 选择 `main` 只是使用最新 workflow 定义；
  - 真正部署对象由 `release_tag` 决定。
- 增加或整理生产 smoke test 步骤，减少每次发布后手工记忆成本。
- 评估是否在 API health 或 CLI `whoami` 中暴露当前服务端版本 / git sha，便于确认线上版本。

验收标准：

- 能从 GitHub Actions 使用 `release_tag=v0.1.2` 部署服务端。
- 部署后能确认线上 `gitSha` / 版本号与 tag 一致。
- 发布手册可以让未来自己按步骤完成部署和回滚。

### P1：微信小程序账号体验

- 新增正式的“个人资料设置”入口：
  - 从“我的”页进入；
  - 修改头像和昵称；
  - 不再通过点击头像/昵称本身触发编辑；
  - 开发环境的“重新设置头像昵称”入口可逐步收敛到设置页。
- 继续保持首次登录资料确认逻辑：
  - 首次登录时引导确认头像和昵称；
  - 后续登录不重复打扰；
  - 未设置昵称时使用稳定默认值。
- 检查头像显示规格：
  - “我的”页顶部头像和设置页头像样式一致；
  - 避免椭圆、拉伸或裁剪异常。

验收标准：

- 新用户首次登录能确认头像昵称。
- 老用户登录不重复弹资料确认。
- 用户能进入设置页修改头像昵称。
- 不新增手机号、定位、相册、通讯录等隐私能力。

### P1：创建表单体验

- 不提供显式“从剪切板导入”按钮，用户按微信原生习惯在输入框内自行粘贴。
- 备注 / 描述默认折叠：
  - 默认只显示“添加”或“已填写”；
  - 点击后展开多行输入区域；
  - 避免低频字段长期占用页面空间。
- 日期和时间合并为同一行：
  - 提醒：截止日期和截止时间放在同一行；
  - 日历：开始日期/时间、结束日期/时间各放在同一行。

验收标准：

- 页面中没有突出的剪切板读取入口。
- 用户可以在标题、备注、描述等输入框中正常粘贴。
- 备注 / 描述默认不展示大面积输入区。
- 日期和时间不再拆成两个独立表单行。

### P1：提醒列表小体验

- 复查提醒项左侧完成圆环的垂直居中。
- 复查左滑删除按钮在不同宽度设备上的边界、圆角和动画表现。
- 删除继续保持二次确认；暂不展示“已删除”列表。

验收标准：

- iPhone 常见宽度下列表不溢出、不露边、不跳动。
- 删除交互有明确分界和确认，不误删。

### P2：CLI / Agent 体验

- 梳理 CLI 常用 smoke 命令：
  - `whoami`
  - `today`
  - `reminder list`
  - `calendar list`
  - `contact list/search/show`
- 评估是否增加一个轻量 smoke 命令或文档化脚本，用于发布后快速验收。
- 保持 PAT 明文仅创建时返回，服务端只存 hash。

验收标准：

- 生产验收流程可在 5 分钟内跑完。
- 测试数据可被清理，不污染真实账号。

### P2：提醒触达准备

`0.1.2` 不强制上线真实订阅消息投递，但需要明确下一步：

- 确认微信订阅消息模板 ID 和模板字段。
- 确认 worker / 队列的生产运行方式。
- 确认用户授权场景和隐私报备是否已覆盖。

验收标准：

- 如果 `0.1.2` 不启用真实触达，则 worker 不作为生产必启进程。
- 如果临时决定启用真实触达，必须先更新隐私报备和发布说明。

## 明确不做

`0.1.2` 暂不做以下能力，避免补丁版本膨胀：

- 提醒详情页。
- 提醒星标。
- 撤销删除或已删除列表。
- 读取手机通讯录。
- 获取手机号。
- 获取地理位置。
- 图片、附件、语音上传。
- 蓝绿 / 灰度发布完整实现。
- MCP Server。

这些能力可以进入 `0.2.0` 或 `0.3.0` 规划。

## 隐私与合规门禁

`0.1.2` 的默认原则是不新增隐私类目。允许继续使用：

- 微信登录标识。
- 用户头像、昵称。
- 用户主动输入的提醒、日历、联系人内容。
- 订阅消息授权结果。
- 用户主动触发的剪贴板写入，以及用户在输入框内自行粘贴的文本。

如果开发中出现以下接口或能力，必须先暂停实现，更新 [privacy-compliance.md](../privacy-compliance.md)，并在微信公众平台完成对应报备：

- `getPhoneNumber`
- `getLocation`
- `chooseImage` / `chooseMedia`
- `getClipboardData`、自动读取剪切板，或将剪切板文本用于提醒、日历创建以外的用途
- 读取手机通讯录
- 上传头像、图片、附件、语音到服务端或对象存储

发布前固定检查：

```bash
rg "getPhoneNumber|chooseImage|chooseMedia|getLocation|getClipboardData|getUserProfile|requestSubscribeMessage|chooseAvatar|type=\"nickname\"" apps/miniapp/miniprogram
```

## 建议实施顺序

1. 完成 `v0.1.1` 服务端 CD 和生产 smoke test。
2. 创建 `0.1.2` 开发分支或继续 main 小步提交。
3. 先做个人资料设置页和头像昵称体验收敛。
4. 复查提醒列表删除和布局小问题。
5. 补发布后 smoke test 文档或脚本。
6. 跑隐私接口检查和全量发布前验证。
7. 生成 `v0.1.2` 发布说明，打 tag，上传小程序，手动 CD。

## 发布备注草案

`0.1.2` 是 ai-todo 内测稳定修订版，重点优化微信小程序账号资料设置体验、创建提醒/日历表单体验、提醒列表交互细节和服务端发布验收流程。版本继续保持小程序客户端、FastAPI 服务端和 CLI 一致发布；生产 CD 使用发布 tag 作为部署入口，避免服务端发布被 main 后续开发提交影响。本版本不新增手机号、定位、相册、通讯录等隐私能力。
