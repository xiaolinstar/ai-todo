## 背景：v0.8.2 做了什么、没做什么

**v0.8.2 已解决**（不再重复进本批）：

- 体验版 / 正式版 / 开发版 **token、隐私同意按 API 环境分桶**
- 模拟器 → 本地 API；手机预览/真机调试/体验版 → staging
- 关于页展示运行环境与 API 地址

**v0.8.2 原计划但未做**（全部划入 v0.8.3）：

- 长时间未打开后的 **静默续登** 与 **会话续期**
- `ensureAuth` / `mine.refreshSession` **竞态** 与误清 token
- 全局 `request` 对 401 的 **一次重试**
- **CLI 移除 `ai-todo token` 子命令**（PAT 管理归小程序 UI）
- 微信提醒关闭 **文案对齐**

## 问题陈述（用户可见 vs 技术根因）

| # | 用户现象 | 当前实现 | 本批目标 |
|---|----------|----------|----------|
| 1 | 隔几周再打开，「我的」显示未登录，需手点「微信登录」 | session token **固定 30 天** `expires_at`，**不随使用延长**；`resolve_token` 只更新 `last_used_at`，不改 `expires_at` | API 滑动续期 + 小程序过期后 **先静默 `wx.login`** |
| 2 | token 刚失效时，一进「我的」立刻变未登录 | `mine.refreshSession`：`fetchMe` 失败 → **立即 `clearToken`**，未尝试续登 | 统一 `restoreSession()`，失败链：静默续登 → 再清 token |
| 3 | 冷启动后其他 Tab 有数据，「我的」仍显示未登录 | `app.onLaunch` 的 `ensureAuth()` 与 `mine.onShow` 的 `refreshSession` **并行、无广播** | 单一会话恢复入口 + 轻量事件通知各页 |
| 4 | 业务请求 401 时只看到「请先登录」，无自动恢复 | `lib/api.ts` **无** 401 重试；各页各自处理 | session 401 → 静默续登 **一次** → 重试原请求 |
| 5 | 关掉「微信提醒」仍可能收到推送，用户困惑 | Worker **不**读 `wechatEnabled`；开关只影响新建/编辑时的订阅授权 | **文案**说明「已授权待送达」vs「不再新授权」 |

（原 v0.8.2 候选「CLI token list 表格化」已取消——PAT 列表改由小程序 Agent 令牌页提供，CLI 不提供 `token` 子命令。）

## 产品决策（实施默认）

| # | 问题 | 默认决策 |
|---|------|----------|
| 1 | 会话续期策略 | **滑动续期**：每次 session token 认证成功，将 `expires_at` 设为 `now + session_token_ttl_days`（默认仍 30 天）；可选 `session_token_max_ttl_days`（如 180）防止从创建日起无限续 |
| 2 | token 绝对过期后 | 小程序 **先静默 `wx.login`** 换新 token；仍失败再展示未登录；**不在首轮 401 就清本地 token** |
| 3 | 启动时是否自动 `wx.login` | **是**（通过 `restoreSession`）：有 token 且失效时静默续登；**无 token** 时预览/正式版仍不主动弹授权（与 v0.8.1 一致，避免无意义消耗 code） |
| 4 | 续登成功是否 toast | **否**（静默）；仅用户主动点「微信登录」时 toast「已登录」 |
| 5 | PAT / CLI session 拒绝 | **不变**；续期仅 `token_type=session` |
| 6 | 关闭微信提醒的行为 | 本批 **仅文案**；批量 skip pending delivery → v0.9 前 backlog |
| 7 | PAT 管理入口 | **仅小程序 UI**；CLI **移除** `ai-todo token`（含 list/create/revoke） |
| 8 | CLI 是否捆绑 tag | 移除 `token` 时可发 CLI `0.6.1`；可与小程序/API 同 tag 或稍后单独 `npm publish` |
| 9 | `login --issue-pat` | **保留**，仅本地 `allow_dev_auth` 开发用 |

## P0 — 登录保活（详细需求）

### REQ-0.1 API：Session Token 滑动续期

**用户价值**：活跃用户长期打开小程序，不必因 30 天绝对过期而频繁重登。

**当前行为**（`apps/api/src/ai_todo_api/modules/api_tokens/service.py`）：

- 创建 session：`expires_at = now + session_token_ttl_days`（默认 30）
- `resolve_token`：校验 `expires_at`、`max_idle_days`（session 为 `None`），更新 `last_used_at`，**不修改 `expires_at`**

**期望行为**：

1. 当 `token_type == session` 且 token 未过期、未 revoke 时，在 `resolve_token` 成功路径上：
   - 将 `expires_at` 更新为 `now + session_token_ttl_days`
   - 若配置了 `session_token_max_ttl_days`，则 `expires_at` 不得超过 `created_at + session_token_max_ttl_days`
2. PAT 行为不变（不续期 `expires_at`，仍受 `max_idle_days` 约束）
3. 配置项（`apps/api/src/ai_todo_api/config.py`）：
   - 保留 `session_token_ttl_days`（默认 30）
   - 新增可选 `session_token_max_ttl_days: int | None`（默认 `None` 或 180，实施时二选一写进 plan）

**验收**：

- [ ] 登录后第 29 天有一次 API 请求 → token 仍有效，且 `expires_at` 延后约 30 天
- [ ] 超过绝对 `expires_at` 且未续期过的 token → 401
- [ ] PAT 创建/使用/idle 逻辑回归通过
- [ ] `pnpm test:api` 新增/更新 `resolve_token` 续期用例

**涉及文件**：`config.py`、`modules/api_tokens/service.py`、`tests/auth/`、`docs/api-design.md`（续期语义）

---

### REQ-0.2 小程序：统一 `restoreSession()`

**用户价值**：任意入口（启动、进「我的」、401 重试）用同一条逻辑恢复登录，减少误清 token 和 UI 不一致。

**当前行为**（`lib/auth.ts`、`pages/mine/mine.ts`）：

- `ensureAuth()`：有 token → `fetchMe`；失败 → `performWechatLogin()`（**会静默 wx.login**），但无 token 时 preview/release **直接 false**
- `hasValidSession()` / `refreshSession()`：`fetchMe` 失败 → **clearToken** + reset UI
- 两处逻辑不一致；`ensureAuth` 成功不通知「我的」页

**期望行为** — 新增 `restoreSession(options?: { silent?: boolean })`：

```text
1. syncRuntimeConfig()（对齐 apiUrl / scope）
2. 若无 token：
   - develop + allow_dev_auth → fetchMe，成功即 loggedIn
   - 否则 → 返回 { ok: false, reason: 'no_token' }
3. 若有 token：
   a. fetchMe 成功 → 返回 { ok: true, user }
   b. fetchMe 失败（401/网络）→ silentWechatReLogin()：
      - wx.login → POST /v1/auth/wechat/login → saveConfig token
      - 再 fetchMe；成功 → { ok: true, user }；失败 → { ok: false, reason: 'relogin_failed' }
4. 仅当 reason === 'relogin_failed' 且确认非网络 transient 时 clearToken()
5. silent === true 时不 toast；用户主动登录仍 toast
```

**调用点**：

| 位置 | 变更 |
|------|------|
| `app.onLaunch` | `restoreSession({ silent: true })` 替代 `ensureAuth()` |
| `mine.onShow` | `restoreSession({ silent: true })` → `applyUser` / `resetProfile` |
| `lib/api.ts` | 401 重试路径调用（见 REQ-0.3） |

**验收**：

- [ ] 有效 token 冷启动 → 「我的」无需等待即显示已登录
- [ ] 过期 token + 微信可 login → 静默续登，无 toast，数据可访问
- [ ] 彻底无效（用户被删、微信失败）→ 未登录文案，**无**原始 Authorization 错误
- [ ] develop + dev auth 不退化

**涉及文件**：`lib/auth.ts`（或新建 `lib/session.ts`）、`app.ts`、`pages/mine/mine.ts`

### REQ-0.3 小程序：401 静默续登并重试一次

**用户价值**：在提醒/日历 Tab 操作时 token 刚过期，自动恢复，不必切到「我的」重登。

**当前行为**：`request()` 收到 401 → 直接 resolve `{ ok: false, error: UNAUTHORIZED }`

**期望行为**：

1. 仅对 **带 Bearer session token** 的请求、且 error 为 `UNAUTHORIZED` / HTTP 401：
   - 若本次 request 尚未重试过 → 调用 `restoreSession({ silent: true })` 中的续登部分
   - 续登成功 → **用新 token 重发原请求一次**
2. 防循环：每个 `request()` 调用最多 **1 次** 重试；重试仍 401 → 返回错误，**不** infinite loop
3. 登录接口 `/v1/auth/wechat/login` 自身 **不参与** 重试
4. 并发：同一时刻多个 401 应 **合并** 为一次 `wx.login`（mutex / shared promise，避免 code 竞争）

**验收**：

- [ ] 模拟过期 token：在提醒列表下拉刷新 → 自动续登并成功加载
- [ ] 续登失败：仍显示「请先登录后继续使用」（v0.7.0 文案）
- [ ] 并发 3 个 API 同时 401 → 仅一次 `wx.login`

**涉及文件**：`lib/api.ts`、`lib/auth.ts`

---

### REQ-0.4 小程序：登录态变更广播

**用户价值**：`onLaunch` 静默续登成功后，用户切到「我的」时不再短暂显示未登录。

**期望行为**：

1. `restoreSession` 成功或确认未登录后，通过 **轻量机制** 通知订阅方，例如：
   - `getApp().globalData.session` + 简单 `onSessionChange` 回调列表，或
   - 微信 `eventChannel` / 自定义 event bus（保持最小实现）
2. **最低要求**：`mine` 页 `onShow` 必须读最新 session 状态（已通过 REQ-0.2 覆盖）
3. **可选增强**：about / settings 子页在 `onShow` 时若检测到 session 变化则刷新

**验收**：

- [ ] 冷启动续登成功 → 首次进入「我的」显示已登录（无 1–2 秒闪未登录）
- [ ] 退出登录（settings-security）→ 各页一致未登录

**涉及文件**：`app.ts`、`lib/auth.ts` 或 `lib/session.ts`

## P1 — 体验收口（详细需求）

### REQ-1.2 CLI：移除 `ai-todo token` 子命令

**用户价值**：PAT 签发与吊销统一在**已微信登录的小程序**完成，CLI 只负责「携带 PAT 调 API」，降低终端误操作与多入口混乱。

**当前行为**（`apps/cli/src/index.ts`）：

```text
ai-todo token list
ai-todo token create --name ...
ai-todo token revoke <token_id>
ai-todo token revoke-all
```

**期望行为**：

1. **删除**上述 `token` 子命令及 `apps/cli/src/commands/token.ts` 的路由注册。
2. **保留**：
   - `ai-todo login --token <pat>` — 粘贴小程序复制的 PAT 写入 `~/.ai-todo/settings.json`
   - `ai-todo login --issue-pat` — 仅本地 `AI_TODO_ALLOW_DEV_AUTH=true` 时一键签发（开发用）
   - `ai-todo logout` — 仅清本机配置，不吊销服务端 PAT
3. `ai-todo help`、[`cli-design.md`](../cli-design.md)、[`apps/cli/README.md`](../../apps/cli/README.md) 与 [agent-usage.md](../agent-usage.md) 同步：PAT 创建/列表/吊销 → **我的 → Agent 令牌**。
4. 用户执行已移除命令时，输出明确引导（例如 `Use the WeChat miniapp: Mine → CLI / Agent tokens`），**不**静默失败。

**小程序侧（已具备，本批不重复开发）**：

| 操作 | 页面 |
|------|------|
| 创建 | `settings-agent` / `settings-token-create` |
| 列表 | `settings-agent` |
| 吊销 | `settings-token-detail` |
| 批量吊销 | `settings-dev`（develop） |

**验收**：

- [ ] `ai-todo token` 任意子命令不再存在（或返回引导文案 + 非零退出码）
- [ ] `ai-todo login --token` / `--issue-pat` / `logout` 行为不变
- [ ] `ai-todo help` 无 `token` 段
- [ ] 文档与 help 一致

**涉及文件**：`apps/cli/src/index.ts`、`apps/cli/src/commands/token.ts`（删除或弃用）、`help.ts`、`README.md`、`docs/cli-design.md`

**版本**：移除 `token` 时 CLI bump **`0.6.1`**（破坏性变更：删除子命令）。

### REQ-1.3 微信提醒关闭说明文案

**用户价值**：用户理解「关开关」与「仍收到旧授权推送」的区别，减少误以为 bug。

**当前行为**：

- 设置页副标题：「创建或修改提醒、日历时自动请求微信提醒」
- 未说明关闭后 **已订阅待送达** 仍可能收到

**期望行为**：

1. 「微信提醒」开关 **关闭** 时，在开关下方或副标题增加说明，例如：
   - 「关闭后，新建或编辑事项不再请求微信提醒授权；此前已授权的提醒仍可能送达。」
2. （可选）「最近送达」子页增加一行引导：「关闭微信提醒不会取消已在队列中的推送。」
3. **不**改 worker 行为（P2）

**验收**：

- [ ] 关闭开关后文案可见、语义清晰
- [ ] 开关打开时文案不冗长

**涉及文件**：`pages/notification-settings/notification-settings.wxml`、`notification-settings.ts`（若需动态文案）
