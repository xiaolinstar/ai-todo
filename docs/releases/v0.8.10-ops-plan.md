# v0.8.10 运维优化规划 — CI/CD 与 DB Migration 解耦

日期：2026-07-11
状态：实施中
目标：K8s 应用发布不再隐式管理 PostgreSQL，也不在 API 启动路径自动执行 Alembic migration。

## 背景

Production 已迁到 K8s 后，API / worker 属于无状态应用，PostgreSQL / PVC 属于有状态生产资产。此前 K8s app overlay 同时包含 Postgres，且 API entrypoint 默认执行 `alembic upgrade head`，导致应用发布、应用回滚和数据库 schema 演进耦合过紧。

## 交付范围

- K8s app overlay 只管理 API、worker、API Service、ConfigMap、Secret。
- PostgreSQL / PVC 拆到独立 db overlay，并使用 StatefulSet 表达有状态语义。
- K8s app 默认 `AI_TODO_SKIP_MIGRATIONS=true`。
- 新增 `db-migration` Job，以指定 release 的 API 镜像 digest 显式执行 `alembic upgrade head`。
- 新增 `DB Migration (K8s)` 手动 workflow，要求生产迁移前确认备份。
- 更新部署、CI/CD、数据库迁移和 PostgreSQL runbook 文档。

## 不在本版本做

- 不迁移到外部托管 PostgreSQL。
- 不自动执行 contract / backfill。
- 不在普通 `CD (K8s)` 中 prune 旧 DB 资源。
- 不在本次代码变更中实际操作线上 Postgres Deployment → StatefulSet；该动作需要维护窗口。

## 验收

- `kubectl kustomize apps/api/deploy/k8s/overlays/production`
- `kubectl kustomize apps/api/deploy/k8s/overlays/production-db`
- `kubectl kustomize apps/api/deploy/k8s/overlays/production-migration`
- `prettier --check` 覆盖新增 workflow、K8s YAML 与相关文档

## 运维顺序

1. 生产发版：继续使用 `CD (K8s)`，只发布 app overlay。
2. 含 expand migration 的版本：先备份 production，再运行 `DB Migration (K8s)`，最后运行 `CD (K8s)`。
3. Postgres Deployment → StatefulSet：单独维护窗口执行，确保同一 PVC 不被 Deployment 与 StatefulSet 同时挂载。
