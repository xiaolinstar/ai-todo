# 平台身份与跨用户建联（规划中）

状态：**未排期实现**；勿按微信「加好友」模型做。

## 目标冲突

| 需求                                      | 风险                                          |
| ----------------------------------------- | --------------------------------------------- |
| 知道「平台上的谁」以便协作、关联提醒/日历 | 公开搜人 ≈ 暴露账号存在与资料，隐私压力大     |
| 通讯录仍是「我的私人备注」                | 与平台 ID 混用会让用户困惑 handle vs username |

## 已共识（技术模型，见 [tech-decisions.md](../tech-decisions.md)）

- `User.username`：平台级、全局唯一、可分享的公开标识（未来）。
- `Contact.handle`：本用户通讯录内本地短名；可关联 `linked_user_id`，但**关联方式**未定。

## 明确不做

- 微信式：搜索 → 加好友 → 双向同意 → 才能看资料。
- 本阶段不做：任意用户按 username 搜到他人并一键加通讯录（已实现草案已撤回）。

## 待产品定稿（再写 API/小程序）

1. **发现**：是否允许按 username 查询？返回哪些字段（最小化：仅 displayName 首字？仅「是否存在」）？
2. **建联**：邀请码 / 一次性链接 / 双方确认 / 仅 CLI Agent 代填 `linked_user_id`？
3. **时区**：默认来源（微信地区、注册时选、IP）？是否允许用户改？改了是否影响已有 `dueAt` 展示？
4. **username**：谁可改、改几次、改后旧链接是否失效？

## 建议实现顺序（未来某版，如 0.3.x）

1. 只读：账户时区/username 展示 + 服务端默认值策略。
2. 可选：受控的「邀请关联」而非公开搜索。
3. 最后：若确有需求再考虑有限的 username 解析 API（审计、频控、最小字段）。

实现前需本文件定稿 + 安全评审，再更新 OpenAPI 与小程序交互稿。
