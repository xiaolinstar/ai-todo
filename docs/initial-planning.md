# ai-todo 初始规划

日期：2026-05-16

## 项目目标

项目名称暂定为 `ai-todo`。

做一个 AI 原生的待办日历管理产品，面向微信小程序用户，同时提供 CLI 能力，让用户可以通过 Claude、OpenClaw、Codex 等 AI CLI 管理自己的待办、提醒事项与日历。

核心不是“给待办加一个 AI 输入框”，也不是在本项目内置自然语言解析或 LLM 编排，而是提供稳定、结构化、可审计的提醒、日历、联系人、CLI、Skills 和 MCP 能力，让 Claude、OpenClaw、Codex 等外部 Agent 能可靠调用。

### 产品边界修订：不内置自然语言解析

`ai-todo` 本身专注能力层，不负责“自然语言解析并执行”。

- 用户自然语言理解、意图判断、时间解析、联系人语义消歧由调用方 Agent 完成
- 本项目提供结构化 HTTP API、CLI、Skills、MCP 工具描述、JSON 输出、幂等、审计和权限
- 不规划服务端 `nl/parse`、CLI `parse` / `ask`、内置 LLM、Prompt 编排或模型调用
- Agent 解析完成后，应调用结构化命令，例如 `reminder create --title ... --due ... --contact ...`
- 联系人能力只提供 CRUD、搜索、精确 ID 引用和必要的结构化错误，不做 LLM 语义解析

## 命名约定

`ai-todo` 是项目名、品牌名和 CLI 名称。

这里的 `todo` 是一个更广义的用户心智词，不限定为数据模型里的某一种对象。它可以泛指用户想让 AI 帮忙管理的“要做的事”和“时间安排”，覆盖：

- `Reminder`：需要在一段时间内完成的提醒事项
- `Calendar`：特定时间点或时间段的日程安排
- `Task`：未来可能扩展的更重工作项，但不进入 MVP

因此，命名层级建议是：

- 产品/品牌/CLI：`ai-todo`
- MVP 核心对象：`Reminder` + `Calendar`
- 暂不使用的重对象：`Task`

这样既保留了 `todo` 这个更通用、好理解、适合 CLI 的名字，又不会把产品定义扩大成任务管理工具。

## 产品定义

产品定义为 `Reminder + Calendar + Contacts` 三个轻量级服务。

其中 `Contacts` 不是独立通讯产品，而是 AI 个人事项管理代理的基础上下文能力。它负责帮助系统稳定理解“某个人是谁、有哪些联系方式、适合通过什么渠道联系”，并为后续邮件发送、日程参与人、提醒转发和 Agent 自动沟通提供可靠基础。

由于产品形态是微信小程序，核心应保持轻量、高频、易理解，不定义为更重的 `Task` 任务管理产品。`Task` 通常意味着项目管理、复杂状态、负责人、协作、子任务和流程流转，这不符合 MVP 的产品气质。

### 功能参考原则

`Reminder`、`Calendar` 和 `Contacts` 的能力设计可以参考飞书 / Lark 这类成熟工具，尤其是日程、提醒、参与人、通讯录、时间解析、权限和审计等方向，但 MVP 必须主动简化。

参考 Lark 时应遵循：

- 只借鉴通用能力模型，不照搬企业协作复杂度
- 优先实现个人用户高频场景，例如创建、查询、完成、改期、提醒
- 复杂能力先记录到文档，不进入第一版实现
- CLI 能力优先面向 Agent 稳定调用，而不是覆盖所有人工操作细节
- API 设计要保留未来扩展空间，但数据库和 UI 不为低频功能提前复杂化

第一阶段建议实现：

- 结构化创建提醒事项
- 查询今日提醒和今日日程
- 标记提醒完成
- 修改提醒时间
- 创建简单日历事件
- 查询指定日期日程
- 创建和查询联系人
- 查询联系人，并在提醒/日程中通过联系人 ID 建立关联

暂缓但记录：

- 完整参与人管理
- 会议室 / 地点资源预订
- 重复日程
- 复杂提醒规则
- 跨日程冲突检测
- 第三方日历深度同步
- 组织级权限与协作
- 完整邮件收发系统

### Contacts

`Contacts` 是个人通讯录能力，重点是“通过结构化搜索、联系人 ID 和联系方式管理，让调用方能可靠知道某个人是谁，以及可以如何联系”。

典型例子：

- “提醒我明天给张三发方案”
- “把今天的会议纪要发给 Alice”
- “下周约李雷吃饭”
- “给客户王总发一封邮件确认报价”

`Contacts` 的特点：

- 关注联系人身份、别名和联系方式
- 支持姓名、昵称、公司、职位、邮箱、手机号等基础字段
- 可以被 `Reminder`、`CalendarEvent` 和未来的邮件发送能力引用
- 需要处理重名、缺失联系方式和联系方式过期等情况
- 对 Agent 很关键，因为 Agent 需要用结构化搜索结果和联系人 ID 稳定引用“发给谁”

第一阶段不做复杂企业通讯录，也不做组织架构和权限继承。MVP 只需要支持个人联系人、基础联系方式、联系人搜索和联系人 ID 引用；自然语言中的联系人理解由调用方 Agent 完成。

邮件发送非常依赖通讯录，但邮件发送本身建议作为后续能力独立设计。否则第一版会同时引入邮箱授权、发信风控、模板、退信、附件、安全审计等复杂问题。当前阶段更合适的路径是：先把 `Contacts` 做成底层对象，再在未来增加 `EmailMessage` / `CommunicationLog` 这类发送与审计模型。

### Reminder

`Reminder` 是一件事情的提醒，重点是“这件事需要在一段时间内完成”。

典型例子：

- 完成数学作业
- 回家前去超市购买鸡蛋
- 周五前提交报销
- 晚上记得给家人打电话

`Reminder` 的特点：

- 关注事情本身
- 关注是否完成
- 可以有截止时间或提醒时间
- 不一定占用一段明确的日历时间
- 适合轻量记录、稍后提醒、完成勾选

### Calendar

`Calendar` 是特定时间点或时间段的安排，重点是“在这个时间执行或参加某件事”。

典型例子：

- 今天 14:00 开会
- 周六 10:00 参观博物馆
- 明天 19:00 和朋友吃饭
- 下周一 09:30 体检

`Calendar` 的特点：

- 关注时间点或时间段
- 通常有明确开始时间，必要时有结束时间
- 可以设定提前提醒，例如提前一天、提前一小时、提前十分钟
- 关注日程冲突、当天安排和时间占用
- 不一定需要完成状态

### 核心区别

`Reminder` 回答的是：“有什么事情需要我记得完成？”

`Calendar` 回答的是：“我在某个时间要做什么？”

因此：

- `Reminder` 管事情和完成
- `Calendar` 管时间和安排
- 两者都可以提醒用户，但提醒的语义不同
- 微信小程序第一阶段只做这两个轻量对象，不引入 `Task`

## 概念边界

### 待办 / 提醒事项 Todo / Reminder

待办更像 Apple 产品里的“提醒事项”，是用户的轻量行动清单，强调快速记录、适时提醒和完成状态。

典型例子：

- 买牛奶
- 明天提交周报
- 给客户发方案
- 记得预约体检

待办的特点：

- 可以没有明确开始时间
- 通常有完成/未完成状态
- 可以有截止时间、优先级、标签
- 更适合快速捕捉和日常执行
- 可以提醒用户，但不一定占用一段日历时间

### 任务 Task

任务比待办更重，通常用于承载一个有上下文、有步骤、有结果要求的工作项，更接近项目管理或团队协作工具里的概念。

典型例子：

- 完成小程序 MVP 需求文档
- 推进客户 A 的合同签署
- 准备下周产品评审

任务的特点：

- 可能包含子任务、描述、附件、负责人、项目归属
- 可能有阶段状态，例如未开始、进行中、阻塞、完成
- 可能需要 AI 拆解、总结、延期建议
- 更适合项目型、工作型事项

### 是否都要做

不建议 MVP 同时做“待办”和“任务”。作为一款轻量级微信小程序，第一阶段应该明确选择“待办/提醒事项”，而不是做完整任务管理。

原因：

- 用户更容易理解“提醒事项”和“日历”
- 微信小程序更适合轻量、高频、随手记录
- “任务”容易把产品带向项目管理、协作、复杂状态流
- AI 原生能力可以先服务于创建、提醒、安排和复盘，不必一开始承载复杂任务管理

建议：

- 产品语言只使用“待办”或“提醒事项”
- CLI 命令也围绕 `todo` / `reminder` 设计
- 底层模型优先命名为 `Reminder` 或 `Todo`
- 暂不暴露 `Task` 概念
- 未来如果需要支持复杂工作项，可以在待办上增加子项、备注、清单等能力，而不是过早引入完整任务系统

## 日历与待办的区别

日历强调“时间占用”，待办强调“事情完成”。

### 日历 Calendar Event

日历事件通常有明确的开始时间和结束时间，会占用一段时间。

典型例子：

- 今天 14:00-15:00 产品评审会
- 周五 10:00-11:30 客户沟通
- 明天 20:00-21:00 健身

日历的特点：

- 有 `start_at` 和 `end_at`
- 关注时间冲突、空闲时间、提醒
- 可以关联地点、参与人、会议链接
- 不一定有“完成”状态

### 待办 Todo

待办可以有截止时间，但不一定占用日历时间。

典型例子：

- 周五前交方案
- 今天记得报销
- 看完某篇文章

待办的特点：

- 可以只有 `due_at`
- 可以没有具体执行时间
- 关注是否完成
- 可以被安排到日历中

### 二者关系

一个待办可以被安排成一个或多个日历块。

例如：

- 待办：完成 MVP 需求文档，周五前完成
- 日历安排：
  - 周三 20:00-21:00 梳理结构
  - 周四 10:00-11:30 写初稿
  - 周五 09:30-10:00 最终检查

因此建议：

- `Todo` / `Reminder` 管事情
- `CalendarEvent` 管时间
- 两者通过 `todo_id` 或 `reminder_id` 关联

## CLI 设计初步方向

CLI 是这个产品 AI 原生能力的关键入口。它既要给人直接使用，也要给 AI Agent 稳定调用。

### 设计原则

- 命令语义清晰，适合人类输入
- 输出支持结构化 JSON，适合 AI CLI 消费
- 所有写操作默认可追踪，必要时可撤销
- 只提供结构化入口，不内置自然语言解析
- CLI 不直接持有微信登录态，使用独立 API Token

### 初始命令草案

```bash
ai-todo login
ai-todo reminder create --title "给客户发方案" --due "2026-05-16T15:00:00+08:00"
ai-todo today
ai-todo list --status pending
ai-todo done <reminder_id>
ai-todo update <reminder_id> --title "给客户发送最终方案"
ai-todo reschedule <reminder_id> --due "2026-05-25T10:00:00+08:00"
ai-todo calendar today
ai-todo plan today
ai-todo contact search "王总"
```

### 面向 AI CLI 的结构化输出

所有查询命令需要支持：

```bash
ai-todo today --json
ai-todo list --json
ai-todo calendar today --json
```

写操作返回结构化结果：

```json
{
  "ok": true,
  "operation": "create_todo",
  "todo": {
    "id": "todo_123",
    "title": "给客户发方案",
    "due_at": "2026-05-16T15:00:00+08:00",
    "status": "pending"
  }
}
```

### 待讨论的 CLI 问题

- CLI 名称暂定为 `ai-todo`
- 是否提供 MCP Server，方便 Claude Desktop / OpenClaw 直接作为工具调用
- 是否所有 AI 写操作都需要二次确认
- CLI 是否支持离线缓存
- 如何支持批量操作与撤销
- 如何设计 Skills / MCP，让外部 Agent 更稳定地调用结构化能力

## 权限与用户信息安全

权限和用户信息安全需要作为专门设计项处理。MVP 可以先实现基础能力，但文档和数据模型要提前保留安全边界，避免后续 Agent、CLI 和原生 App 接入时难以收敛。

### 用户登录

第一阶段建议：

- 微信小程序使用微信登录换取服务端用户身份
- 服务端维护统一 `User`，不要让业务逻辑直接依赖微信 openid
- CLI 不复用微信登录态，使用独立 API Token
- Token 创建、吊销、过期时间和最后使用时间需要可追踪

后续多端登录需要统一到账号体系：

- 微信登录
- Apple 登录
- 手机号 / 邮箱登录
- 第三方 OAuth 登录
- 多设备会话管理

MVP 可以先只做微信登录和 CLI Token，但数据模型应能支持多登录方式。

### CLI 授权

CLI 是 Agent 调用产品能力的关键入口，不能简单等同于网页登录。

建议设计：

- 用户在小程序或 Web 控制台中创建 CLI Token
- Token 只展示一次，服务端只保存哈希
- Token 支持作用域，例如 `read`, `write`, `calendar:write`, `agent:plan`
- Token 支持过期时间
- Token 支持按设备或用途命名，例如 `MacBook Codex`
- 所有 CLI 写操作写入 `CommandLog`
- 高风险操作需要二次确认或显式参数，例如批量删除、批量改期、长时间自动规划

### Agent 调用边界

Agent 可以帮助用户规划，但必须被限制在明确授权范围内。

需要考虑：

- 每个 Agent Token 的读写权限
- 单次请求可修改的最大对象数量
- 每分钟 / 每小时请求限制
- 写操作幂等键，避免 Agent 重试导致重复创建
- 操作审计日志，记录来源、参数摘要、影响对象和结果
- 可撤销操作设计，例如最近一次批量改期可回滚
- 对外部 Agent 生成的计划保留来源、参数摘要和操作审计，解释文本由调用方负责

### 防滥用与 DDoS

对 Agent 开放 API 后，需要从 MVP 阶段就规划防滥用能力。

MVP 阶段至少设计：

- API Token 级别 rate limit
- 用户级别 rate limit
- IP 级别 rate limit
- 高成本写接口、批量接口和导入接口单独限流
- 写接口限制请求体大小和批量数量
- 失败登录、失败 Token 验证的频率限制
- CommandLog 记录异常来源，便于后续封禁和风控

后续增强：

- 风险评分
- 异常行为检测
- WAF / CDN 层防护
- 队列削峰
- Agent 专用网关
- 按作用域、请求频率、批量操作规模和用户套餐进行额度管理

### 数据安全原则

默认原则：

- 最小权限
- Token 不明文落库
- 敏感日志脱敏
- 用户数据按 `user_id` 强隔离
- 删除和导出能力预留
- Agent / CLI 只读取完成当前请求所需的最小上下文
- 面向 Agent 的 JSON 输出避免泄露不必要的用户资料

## MVP 范围建议

第一阶段先完成最小闭环：

- 微信小程序登录
- 创建待办
- 今日待办列表
- 标记完成
- 简单日历视图
- 创建和查询联系人
- 通过联系人 ID 关联提醒和日程
- 结构化创建待办
- CLI 登录
- CLI 创建、查询、完成待办
- CLI 查询联系人
- 服务端统一 API

暂缓：

- 完整项目管理
- 多人协作
- 完整邮件发送和收件箱管理
- 复杂工作流自动化
- 深度第三方日历同步
- 高级 AI 自动排期
- 完整任务管理

## 初始技术方向

长期产品形态不应只按微信小程序设计。微信小程序适合作为 MVP 的低摩擦入口，但后期计划演进出 iOS、鸿蒙、安卓原生版本，因此项目骨架从一开始要按多端产品规划。

原生 App 的价值：

- iOS 可以使用系统通知，提醒可靠性和触达效果优于小程序
- iOS 可以结合灵动岛、锁屏组件、小组件等系统级入口
- 安卓和鸿蒙可以使用各自系统通知、桌面组件和后台提醒能力
- 原生端更适合承载长期个人助理体验，例如持续提醒、离线缓存和本地快速录入

多端架构建议：

- 小程序端：作为 MVP 首发端，验证核心使用场景
- iOS 原生端：后续重点支持系统通知、灵动岛、锁屏组件和快捷操作
- Android 原生端：后续支持系统通知、桌面组件和后台任务
- HarmonyOS 原生端：后续支持鸿蒙生态通知和服务卡片
- CLI：作为 AI Agent 和高级用户入口
- 后端 API：作为所有客户端和 Agent 的统一能力层

推荐 monorepo 骨架：

```text
apps/
  api/              # 后端 API 服务
  miniapp/          # 微信小程序 MVP
  cli/              # ai-todo CLI
  ios/              # 预留 iOS 原生 App
  android/          # 预留 Android 原生 App
  harmony/          # 预留 HarmonyOS 原生 App
packages/
  shared/           # 共享类型、schema、错误码
  api-client/       # 多端复用 API Client
  agent-protocol/   # CLI / MCP / Agent 工具协议
  config/           # 共享配置、lint、tsconfig
docs/
  cli-design.md
  api-design.md
  data-model.md
  security-design.md
  mobile-roadmap.md
```

可选架构：

- 小程序端：微信小程序原生 / Taro / uni-app
- 后端：Python / FastAPI
- 数据库：PostgreSQL
- ORM：SQLAlchemy
- Migration：Alembic
- CLI：Node.js / TypeScript，方便通过 npm 分发
- Agent 集成：Skills / MCP / 结构化 HTTP API，不在服务端封装模型调用

如果 MVP 主要目标是尽快验证产品闭环，建议第一阶段优先选择：

- 小程序端：微信小程序原生或 Taro
- 后端：Python / FastAPI
- CLI：Node.js / TypeScript
- 数据库：PostgreSQL
- ORM：SQLAlchemy 2.x + Alembic

原生 App 不在 MVP 立刻实现，但目录和文档可以先预留，避免后续 API、Token、通知模型只围绕小程序设计。

核心数据对象：

- `User`
- `Todo` / `Reminder`
- `CalendarEvent`
- `Contact`
- `ContactMethod`
- `Tag`
- `CommandLog`
- `ApiToken`
- 后续可扩展：`EmailMessage` / `CommunicationLog`

## 近期推进计划

### 第 1 步：产品概念定稿

- 明确待办、任务、日历的产品边界
- 明确通讯录是基础上下文能力，不是独立社交通讯产品
- 确认 MVP 只暴露“待办/提醒事项”、“日历”和“联系人”
- 确认第一阶段不做完整“任务”概念

### 第 2 步：CLI 详细设计

- 定义命令列表
- 定义 JSON 输出格式
- 定义错误码
- 定义登录和 Token 机制
- 定义 AI Agent 调用约束
- 定义 Token 作用域、过期、吊销和审计日志
- 定义 Agent 写操作确认与幂等规则
- 定义联系人搜索、候选展示和联系人 ID 引用规则

### 第 3 步：数据模型与 API 设计

- 设计数据库 schema
- 设计 REST/tRPC API
- 设计联系人、联系方式和联系人引用模型
- 设计审计日志和撤销机制
- 设计基础限流、防滥用和 DDoS 防护策略
- 参考 Lark 的日程、提醒和通讯录能力，但只纳入 MVP 必需字段

### 第 4 步：项目骨架开工

- 初始化 monorepo
- 创建小程序应用目录
- 创建后端服务目录
- 创建 CLI package
- 建立共享类型包
- 预留 iOS、Android、HarmonyOS 原生 App 目录或规划文档
- 建立共享 API Client 和 Agent 协议包

## 今日结论

今天先把方向记录下来：这个产品应更像 Apple 的“提醒事项 + 日历”，并补充个人通讯录作为 Agent 和未来邮件发送的基础能力，而不是项目管理工具。MVP 应以“待办/提醒事项”、“日历”和“联系人”作为核心入口，暂不引入完整“任务”概念。功能能力可以参考 Lark 等成熟工具，但第一阶段必须主动简化。ai-todo 自身只提供结构化 CLI、Skills、MCP、HTTP API、联系人搜索、权限、审计和持久化；自然语言理解、意图识别和执行规划交给 Claude、OpenClaw、Codex 等调用方。

下一次可以重点讨论 CLI 的命令体系、Agent 工具协议和安全授权模型，然后再进入项目骨架搭建。
